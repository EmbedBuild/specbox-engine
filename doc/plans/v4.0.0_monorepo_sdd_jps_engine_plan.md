# Plan: v4.0.0 — Monorepo SDD-JPS Engine

> Generado: 2026-03-08
> Engine actual: v3.9.1
> Version objetivo: v4.0.0 "Monorepo + Gherkin Acceptance"

---

## Resumen

Fusión de 3 repositorios en un monorepo `sdd-jps-engine`, rename completo,
integración de tools de Trello de dominio en el MCP del engine, y adopción
de Gherkin en español como mecanismo estándar de validación de Acceptance Criteria.

---

## Decisiones firmes (no negociables)

| Decisión | Detalle |
|----------|---------|
| Vocabulario | US/UC/AC — fijo, hardcoded, sin configuración |
| Gherkin idioma | Español (`# language: es`) en todos los `.feature` |
| Trello | Opt-in por proyecto. `setup_board` crea board nuevo, nunca migra boards manuales |
| Monorepo | Un solo repo, un solo deploy, una sola versión |
| MCP unificado | 73+ tools en un solo endpoint (engine + spec-driven + telemetría) |
| `trello-iautomat` | Fuera de la ecuación. No se usa en el engine |
| `sync_engine` | Eliminado. Innecesario en monorepo |

Proceso:

2. Generar `.feature` en español con tags @US-XX @UC-XXX @AC-XX

5. Capturar screenshots por escenario
6. Generar `results.json` estándar desde JSON Cucumber
7. Commit: `git add test/acceptance/`

#### 5.2 Adaptar agents/acceptance-validator.md (AG-09b)

Input:
- `.feature` files generados
- JSON Cucumber report (resultado de ejecución)
- Screenshots en `.quality/evidence/{feature}/acceptance/`
- Código implementado (git diff)

Validación por AC:
1. ¿El `.feature` tiene un @AC-XX Escenario para este criterio? → CHECK
2. ¿El JSON report muestra PASSED para este Escenario? → CHECK
3. ¿Existe screenshot/evidencia visual? → CHECK
4. ¿El código implementado cubre la lógica del criterio? → CHECK (git diff)

Output:
- `acceptance-report.json` con veredicto por AC
- `acceptance-report.pdf` con tabla de escenarios + screenshots
- Veredicto: ACCEPTED (100% pass) / CONDITIONAL (≥80%) / REJECTED (<80%)

## BDD / Gherkin — Acceptance Testing
