# SpecBox Engine v5.23.0 — "Full Mutations"

> **Status**: Design locked — implementation pending
> **Branch**: `feature/full-mutations-v5.23.0`
> **Target release**: v5.23.0 (minor, no breaking changes)
> **Tool count**: 114 → **138** (+24)
> **Origin**: Addendum from consulting session 2026-04-15 on `potencial_digital_2026`, where 21 operational gaps blocked the transition from initial setup to daily engine operation.

---

## Purpose

v5.22.x exposes full **read** access and workflow **state transitions** over spec-driven items (`get_uc`, `list_us`, `move_uc`, `mark_ac`, `start_uc`, `complete_uc`, `attach_evidence`, etc.) but no granular **metadata mutations** and no **planning automation**. Teams are forced to either re-run the destructive `import_spec` or hand-edit cards in Trello/Plane, breaking the spec-driven promise.

This release closes that gap. It adds 24 tools across 4 tiers that cover:

1. **Granular metadata mutation** of existing US/UC/AC (with batch variants for the common loops)
2. **Milestone-based budgeting** (H1..H4 payment milestones) and **multirepo/satellite** operations
3. **Board-level operations** for lifecycle, quality, auditing and archival
4. **Acceptance automation** that composes the lower tiers into high-value workflows

The release is **non-breaking**: every existing tool keeps its signature and semantics.

---

## Guiding principles

These are the rules that governed the design. Re-read them before implementing any tool.

### 1. Batch-first over granular

Whenever a real operational case involves N>1 items, the **batch variant is a first-class tool**, not an afterthought. Granular tools exist for genuine 1:1 operations (e.g. `delete_uc` with per-UC `reason`, `link_uc_parent`). Every granular tool's docstring must reference its batch equivalent in the first paragraph so the LLM discovers it at tool-selection time.

This principle is saved to engram as `architecture/mcp-batch-first` and applies to all future tool design in this repo.

### 2. Reuse `SpecBackend.update_item(meta=...)` — don't grow the ABC

`SpecBackend.update_item` already accepts a `meta: dict[str, Any] | None` kwarg with *only-non-None-changes* semantics. All three backends (Trello via custom fields, Plane via description_html regex rebuild, FreeForm via JSON) already implement it correctly.

**Therefore:** Tier 1 introduces **zero** new abstract methods in the ABC. Every metadata update is a thin tool-layer wrapper over `backend.update_item(meta={...})` with spec-driven validation (milestone ∈ H1..H4, uc_id exists, satellite is a declared key, etc.).

Tiers 2–3 introduce a minimum of new ABC methods, and only where there is genuine backend-specific behavior. Specifically:

- `archive_item` — new ABC method. Each backend has a different archival mechanism (Trello: move to `Archived` list or add `archived` label; Plane: move to `Cancelled` state with comment; FreeForm: move entry to `archive.json`). Cannot be emulated via `update_item`.
- `get_item_comments_since` — optional helper if `get_board_diff` requires it; for v1 we build diff purely from `list_items` snapshots taken by the tool layer itself.

### 3. Validation lives in the tool layer

Spec-driven concepts (milestones, satellites, AC format, link types) are engine-level, not backend-level. Backends know nothing about "H1" or what a "satellite" is. All validation happens in `server/tools/*.py` before the tool dispatches to `backend.update_item`. This keeps backends thin, avoids triplication, and makes testing with mocks trivial.

### 4. Everything is idempotent

Every mutation tool is safe to call twice. If the target state is already the desired state, return `{updated_fields: [], reason: "no_change"}` rather than erroring. This is a hard rule, not a recommendation — AC-19 enforces it.

### 5. Errors are structured, not exceptions

Every tool returns either a success dict or `{error: str, code: str, ...context}`. No raised exceptions to the MCP client. Standard error codes:

- `UC_NOT_FOUND`, `US_NOT_FOUND`, `AC_NOT_FOUND`
- `INVALID_MILESTONE`, `INVALID_SATELLITE`, `INVALID_LINK_TYPE`
- `VALIDATION_FAILED` (with `details: list[{field, issue}]`)
- `BACKEND_ERROR` (with `backend: str, raw: str`)

### 6. Tool selection guidance in docstrings

Every granular tool's docstring opens with one of these two blocks:

```
Use this tool for: <genuine 1:1 operation>.
For batch operations on N UCs/ACs, use: <batch_tool_name>.
For bulk ingest of a new spec, use: import_spec.
```

or

```
Use this tool when: <condition>.
Do NOT use this for: <misuse>. Use <alternative> instead.
```

This matters because the LLM picks tools by reading descriptions. Bad guidance → wasted tokens on granular loops.

## Architecture map

```
server/
├── spec_backend.py              ← SpecBackend ABC (+ 1 new method: archive_item)
├── backends/
│   ├── trello_backend.py        ← +1 method (archive_item via "Archived" list)
│   ├── plane_backend.py         ← +1 method (archive_item via Cancelled state + comment)
│   └── freeform_backend.py      ← +1 method (archive_item via archive.json append)
├── tools/
│   ├── spec_driven.py           ← UNCHANGED. Existing 21 tools.
│   ├── spec_mutations.py        ← NEW. Tier 1 — 8 tools (6 granular + 2 batch)
│   ├── milestone_management.py  ← NEW. Tier 2 — 8 tools (7 + 1 batch)
│   ├── board_operations.py      ← NEW. Tier 3 — 5 tools
│   └── acceptance_automation.py ← NEW. Tier 4 — 3 tools
└── server.py                    ← +4 register_*_tools() calls, tool count 114 → 138
```

**Why 4 modules instead of 1:** keeps each file under ~600 lines, allows per-tier test organization, and lets individual tiers be merged as independent PRs if the release is split.

**Registration order in `server.py`:** after `spec_driven` and `migration`, before `stitch`. Rationale: the new tools extend the spec-driven surface, so they group naturally with it.

# Tier 1 — Granular Mutations (`spec_mutations.py`)

**8 tools**: 6 granular + 2 batch. Zero ABC changes.

All tools take `board_id: str` and `ctx: Context` as first and last args (per existing `spec_driven.py` convention). They call `get_session_backend(ctx)` to resolve the active backend, and always `await backend.close()` in `finally`.

### 1.1 `update_uc`

```python
async def update_uc(
    board_id: str,
    uc_id: str,
    ctx: Context,
    *,
    name: str | None = None,
    description: str | None = None,
    hours: float | None = None,
    screens: list[str] | None = None,
    actor: str | None = None,
    context_text: str | None = None,
    milestone: str | None = None,
    satellite: str | None = None,
) -> dict[str, Any]:
    """Update metadata of a single UC without touching ACs or workflow state.

    For batch updates across many UCs, use `update_uc_batch` — it's 1 MCP
    call instead of N, and keeps token cost proportional to payload size,
    not call count.

    For bulk initial ingest of UCs, use `import_spec` — this tool is for
    point edits after import.

    Only non-None fields are updated (merge semantics). milestone must be
    one of H1, H2, H3, H4. satellite must match a key declared in the
    orchestrator's settings.local.json → multirepo.satellites.

    Returns:
        {
            "uc_id": str,
            "updated_fields": list[str],
            "backend_item_url": str,
            "updated_at": str (ISO 8601 UTC),
        }

    Errors:
        UC_NOT_FOUND, INVALID_MILESTONE, INVALID_SATELLITE, BACKEND_ERROR
    """
```

**Implementation sketch**:

```python
backend = await get_session_backend(ctx)
try:
    uc_item = await find_uc(backend, board_id, uc_id)
    if not uc_item:
        return {"error": f"Use Case {uc_id} not found", "code": "UC_NOT_FOUND"}

    if milestone is not None:
        ok, err = validate_milestone(milestone)
        if not ok: return {"error": err, "code": "INVALID_MILESTONE"}

    if satellite is not None:
        ok, err = validate_satellite(satellite, _settings_path_from_ctx(ctx))
        if not ok: return {"error": err, "code": "INVALID_SATELLITE"}

    meta_updates = {
        k: v for k, v in {
            "horas": hours,
            "pantallas": screens,
            "actor": actor,
            "context": context_text,
            "milestone": milestone,
            "satellite": satellite,
        }.items() if v is not None
    }
    merged_meta, changed = merge_meta(uc_item.meta, meta_updates)

    if not changed and name is None and description is None:
        return {
            "uc_id": uc_id,
            "updated_fields": [],
            "backend_item_url": uc_item.url,
            "updated_at": _utc_now_iso(),
            "reason": "no_change",
        }

    updated = await backend.update_item(
        board_id, uc_item.id,
        name=name, description=description,
        meta=merged_meta if changed else None,
    )
    return {
        "uc_id": uc_id,
        "updated_fields": changed + (["name"] if name else []) + (["description"] if description else []),
        "backend_item_url": updated.url,
        "updated_at": _utc_now_iso(),
    }
finally:
    await backend.close()
```

**Key decisions**:
- `hours` is **float** (not str) for type safety — backend serializes as needed.
- `context_text` is the kwarg name because `context` is already taken by the MCP `Context` parameter.
- Workflow state (`state`) is **not** updatable via `update_uc` — use `move_uc`. This is intentional to keep responsibilities clear.

**AC coverage**: AC-04 (milestone label), AC-19 (idempotency via no_change short-circuit)

### 1.2 `update_uc_batch`

```python
async def update_uc_batch(
    board_id: str,
    updates: list[dict],
    ctx: Context,
    *,
    stop_on_error: bool = False,
) -> dict[str, Any]:
    """Update metadata of many UCs in a single MCP call.

    Each entry in `updates` is a dict:
        {
            "uc_id": str,           # required
            "name": str | None,
            "description": str | None,
            "hours": float | None,
            "screens": list[str] | None,
            "actor": str | None,
            "context_text": str | None,
            "milestone": str | None,
            "satellite": str | None,
        }

    Use this tool when you need to update >3 UCs — it saves MCP round-trips.
    For single-UC edits use `update_uc`. For bulk initial ingest use `import_spec`.

    By default, errors on individual UCs are collected and the batch continues
    (partial success). Set stop_on_error=True to abort on first failure.

    Returns:
        {
            "total": int,
            "succeeded": list[{uc_id, updated_fields}],
            "failed": list[{uc_id, error, code}],
            "backend_item_urls": dict[uc_id, url],
        }
    """
```

**Implementation note**: loads `list_items` **once** and resolves all UCs client-side. This is the big win vs. granular loops — not just N→1 round-trips but also 1 `list_items` call total.

**AC coverage**: AC-19, AC-21 (real case: 27 UCs of potencial_digital_2026 in one call)

### 1.4 `update_ac`

```python
async def update_ac(
    board_id: str,
    uc_id: str,
    ac_id: str,
    ctx: Context,
    *,
    text: str | None = None,
    done: bool | None = None,
) -> dict[str, Any]:
    """Rewrite an AC's text and/or change its done state.

    Distinct from `mark_ac`, which only toggles `done`. Use `update_ac` when
    the Definition Quality Gate flags an AC as vague or untestable and you
    need to rewrite it.

    For batch updates across many ACs, use `update_ac_batch`.

    Only non-None fields are updated. If done is not passed, the existing
    done state is preserved (AC-06).

    Returns:
        {
            "uc_id": str,
            "ac_id": str,
            "updated_fields": list[str],
            "updated_at": str,
        }
    """
```

**Implementation note**: AC text lives in different structures per backend (Trello: checklist item name; Plane: sub-work-item name/description; FreeForm: item in items.json with label AC). The tool layer resolves through `backend.get_acceptance_criteria` + a new helper `backend.update_acceptance_criterion` — **wait**.

**Design decision:** to update AC text we need either:
- (a) a new ABC method `update_acceptance_criterion(board_id, uc_item_id, ac_id, text)`, OR
- (b) resolve the AC's backend ID via `get_acceptance_criteria` and call `update_item` on it (works for Plane and FreeForm where ACs are items, fails for Trello where ACs are checklist items).

**Chosen**: (a) — add **one** new ABC method `update_acceptance_criterion` because Trello's checklist-item path and Plane's sub-item path genuinely differ. This is the only ABC addition in Tier 1 (plus `archive_item` in Tier 3). Updated architecture map above.

**Revised architecture delta**:
```
server/spec_backend.py       +2 methods: update_acceptance_criterion, archive_item
server/backends/*.py         +2 methods each (6 implementations total)
```

**AC coverage**: AC-06

### 1.7 `delete_ac`

```python
async def delete_ac(
    board_id: str,
    uc_id: str,
    ac_id: str,
    ctx: Context,
    *,
    reason: str | None = None,
) -> dict[str, Any]:
    """Remove an AC from a UC and renumber subsequent ACs.

    If reason is provided, it's logged as a comment on the UC card for audit.

    Renumbering: if AC-03 is deleted from [AC-01, AC-02, AC-03, AC-04, AC-05],
    the result is [AC-01, AC-02, AC-03 (was AC-04), AC-04 (was AC-05)].

    Returns:
        {
            "uc_id": str,
            "deleted_ac_id": str,
            "renumbered_acs": dict[old_id, new_id],  # empty if nothing renumbered
            "deleted_at": str,
        }
    """
```

**Implementation complication**: Trello checklist items can be deleted individually, Plane sub-work-items can be deleted, FreeForm items can be removed from items.json. The **renumber** step is the hard part — it needs to update every AC after the deleted one. For Plane/FreeForm this means updating N items. For Trello this means updating N checklist items.

**Decision**: renumbering uses `update_acceptance_criterion` in a loop (batch-sized) over the affected ACs. Not ideal but correct.

**AC coverage**: AC-08

```python
async def bulk_update_hours_from_description(

    and sync to the structured hours field.

    Matched patterns:
    - "Horas estimadas: 8"
    - "Estimación: 8h"
    - "Estimacion: 8h"
    - "Horas: 8"

    dry_run=True returns the list of proposed changes without applying them.
    dry_run=False applies them via update_uc_batch (NOT a loop of update_uc).

            "parsed_ucs": [

```

**Real case**: 22 of 27 UCs on potencial_digital_2026 had hours in description text but not in field.

**Action semantics**:
- `update`: field is None, text has a number → will update
- `skip`: field matches text → nothing to do
- `conflict`: field has value, text has different value → flagged in report, not auto-applied even with dry_run=False (human must decide)

```python
    Returns:
        {
            "uc_id": str,
            "total_acs": int,
            "classified": {"simple": int, "integration": int, "e2e": int},

            "strategy": str,
```

```python
async def milestone_acceptance_check(
```

# Open questions / risks
