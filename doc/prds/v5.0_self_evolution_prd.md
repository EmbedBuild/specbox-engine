# PRD: SDD-JPS Engine v5.0 — Self-Evolution Features

**Proyecto:** sdd-jps-engine  
**Autor:** Jesús Pérez Sánchez (JPS Developer)  
**Fecha:** 2026-03-13  
**Estado:** Draft → Pendiente validación  
**Stack:** Python 3.12+ / FastAPI / FastMCP 3.0.0+  
**Backend de spec tracking:** Trello / Plane (existente, multi-backend desde v4.1.0)

---

## 1. Contexto y Motivación

El análisis comparativo del SDD-JPS Engine v4.2.0 frente al estado del arte del Spec-Driven Development (Kiro, BMAD, Spec Kit, Intent, OpenSpec) identificó 5 áreas de mejora con impacto directo en la productividad diaria del desarrollador y en la posición competitiva del Engine.

Este PRD define las 5 User Stories necesarias para implementar estas mejoras, priorizadas por impacto en el día a día de un desarrollador tipo que gestiona 2-3 proyectos activos con el Engine.

### Audiencia

- **Primaria:** Desarrollador senior/tech lead que usa Claude Code con el Engine a diario.
- **Secundaria:** Nuevos desarrolladores que se incorporen al equipo o adopten el Engine.
- **Terciaria:** Stakeholders no técnicos (como el Financial Controller de McProfit) que consultan PRDs como fuente de verdad.

### Métricas de éxito globales

| Métrica | Baseline actual | Objetivo post-v5.0 |
|---------|----------------|---------------------|
| Tiempo de context recovery (retomar proyecto) | 30-45 min | < 10 min |
| Tiempo de revisión de PR externa contra spec | 20-30 min | < 5 min |
| Días hasta productividad de un nuevo developer | 4-5 días | 1-2 días |
| Ahorro semanal acumulado por developer | — | 3-5 horas |
| Acceptance rate AG-09b en PRs externas | 0% (no disponible) | > 90% |

### Restricciones

- No modificar el flujo core de /prd, /plan, /implement — las mejoras se añaden como extensiones o pasos adicionales.
- Mantener retrocompatibilidad total con proyectos ya onboarded en v4.2.0.
- No introducir dependencias externas nuevas que no estén en pyproject.toml actual.
- Las mejoras deben funcionar tanto con Trello como sin Trello (modo freeform).

### VEG Readiness

No aplica — este PRD es para mejoras internas del Engine, no para un producto con UI de usuario final. No se generan artefactos VEG.

## 2. User Stories

---

### US-01: Spec-Code Sync Layer

**Como** desarrollador que usa el Engine en múltiples proyectos,  
**quiero** que tras cada fase de /implement el PRD se actualice automáticamente con el estado real de la implementación,  
**para** poder retomar cualquier proyecto en minutos consultando un único documento que refleje la verdad del código.

**Prioridad:** P0  
**Estimación:** 11-16 horas  
**Impacto semanal:** Ahorro de 2-3 horas por desarrollador

---

#### UC-001: Generación de Implementation Delta post-fase

**Descripción:** Al finalizar cada fase del /implement (DB, Feature, UI, QA, Integration), el Orchestrator genera un resumen estructurado de lo que se implementó, qué cambió respecto al plan, y qué decisiones se tomaron. Este resumen se almacena en memoria para su posterior escritura en el PRD.

**Pre-condiciones:**
- /implement en ejecución con plan cargado.
- Sub-agente de la fase ha completado su trabajo y devuelto summary al Orchestrator.

**Agentes involucrados:** Orchestrator (lógica de diff), AG-01/AG-02/AG-03 (proveen summaries).

**AC-01:** Tras completar cada fase del /implement, el Orchestrator genera un bloque Markdown con formato predefinido que contiene: nombre de la fase, archivos creados/modificados, cambios respecto al plan (deltas), y decisiones tomadas por el sub-agente.

**AC-02:** El bloque generado no excede 500 tokens por fase para respetar el token budget del Orchestrator (max 15% context window).

**AC-03:** Si la fase no produjo cambios respecto al plan (implementación exacta), el bloque indica "Sin deltas — implementación conforme al plan" en lugar de omitirse.

**AC-04:** Si la fase activó self-healing, el bloque incluye una línea adicional con el tipo de healing aplicado (auto-fix, diagnóstico, rollback) y si se resolvió.

**AC-05:** El bloque se genera incluso si la fase falla y el pipeline continúa al siguiente intento. En ese caso indica "Fase fallida — pendiente de resolución" con el error resumido.

---

#### UC-002: Escritura de Implementation Status en PRD

**Descripción:** Al completar un UC (complete_uc) o al finalizar todos los UCs de una US, el Engine escribe una sección "Implementation Status" al final del PRD correspondiente, compilando todos los deltas de fase generados por UC-001.

**Pre-condiciones:**
- UC-001 ha generado los bloques de delta para todas las fases completadas.
- El PRD existe en doc/prd/ y es accesible por el Orchestrator.

**Agentes involucrados:** Orchestrator (escritura), MCP tools (localización del PRD).

**AC-06:** Al ejecutar complete_uc, el Engine añade una sección `## Implementation Status — UC-{id}` al final del PRD con todos los bloques delta de las fases completadas de ese UC.

**AC-07:** La sección se añade mediante append al final del archivo Markdown, sin modificar el contenido existente del PRD (no reescribe secciones anteriores).

**AC-08:** Si el PRD ya tiene una sección Implementation Status de UCs anteriores, el nuevo UC se añade a continuación manteniendo orden cronológico.

**AC-09:** Si el Engine opera en modo freeform (sin Trello), la escritura se ejecuta igualmente al detectar que el pipeline ha completado todas las fases del plan.

**AC-10:** La sección incluye un timestamp ISO 8601 y el identificador de la rama feature/ desde la que se implementó.

---

#### UC-003: MCP Tool get_implementation_status

**Descripción:** Nueva tool MCP que permite consultar el Implementation Status de cualquier US o UC de un proyecto onboarded, devolviendo los deltas compilados en formato estructurado.

**Pre-condiciones:**
- Proyecto onboarded con PRD que contenga sección Implementation Status.

**Agentes involucrados:** MCP Server (tools/spec_driven.py).

**AC-11:** La tool `get_implementation_status` acepta como parámetro un US-id o UC-id y devuelve el bloque de Implementation Status correspondiente en formato JSON con campos: uc_id, timestamp, branch, phases (array de deltas), overall_status (conforme/con_deltas/parcial).

**AC-12:** Si el UC no tiene Implementation Status (aún no implementado), la tool devuelve un objeto con status "not_implemented" y el campo phases vacío.

**AC-13:** Si se pasa un US-id, la tool devuelve un array con el Implementation Status de todos sus UCs hijos, ordenados por fecha de implementación.

**AC-14:** La tool incluye un campo `delta_count` que indica cuántos cambios respecto al plan se detectaron en total, permitiendo una evaluación rápida de la divergencia spec-code.

### US-02: DX de Onboarding Mejorada

**Como** nuevo desarrollador que se incorpora a un proyecto que usa el Engine,  
**quiero** un flujo guiado que me enseñe el pipeline completo paso a paso con un micro-proyecto de ejemplo,  
**para** ser productivo en 1-2 días en lugar de 4-5.

**Prioridad:** P1  
**Estimación:** 16-24 horas  
**Impacto:** Reducción de tiempo a productividad de 4-5 días a 1-2 días

---

#### UC-004: Skill /quickstart — Tutorial guiado interactivo

**Descripción:** Nuevo Agent Skill que guía al desarrollador por un flujo completo /prd → /plan → /implement usando un micro-proyecto de ejemplo (CRUD de notas). El skill explica cada paso, cada decisión del Engine, y los conceptos fundamentales (US/UC/AC, quality gates, acceptance testing).

**Pre-condiciones:**
- Engine instalado (install.sh completado).
- Claude Code disponible.

**Agentes involucrados:** Nuevo skill /quickstart (archivo .claude/skills/quickstart/SKILL.md).

**AC-15:** El skill /quickstart crea un directorio temporal `~/quickstart-demo/` con un proyecto Flutter mínimo preconfigurado (pubspec.yaml, lib/main.dart básico, .claude/project-config.json con Trello deshabilitado).

**AC-16:** El skill guía al desarrollador por 4 etapas secuenciales: (1) Ejecutar /prd con un prompt predefinido para un CRUD de notas, (2) Revisar el PRD generado explicando cada sección, (3) Ejecutar /plan y explicar las fases, (4) Ejecutar /implement en modo demo (sin PR real, solo muestra el flujo).

**AC-17:** Entre cada etapa, el skill muestra un bloque explicativo de 3-5 líneas que describe qué acaba de pasar y por qué, usando lenguaje no técnico donde sea posible.

**AC-18:** La etapa 4 (/implement) se ejecuta en modo dry-run: el skill simula el flujo mostrando qué haría cada sub-agente, qué quality gates se evaluarían, y qué acceptance criteria se verificarían, sin ejecutar código real.

**AC-19:** Al finalizar, el skill muestra un resumen de los conceptos aprendidos y una referencia a los docs/ del Engine para profundizar. El directorio temporal se ofrece borrar o mantener.

**AC-20:** El skill completo se ejecuta en menos de 5 minutos de interacción activa del desarrollador (excluyendo tiempos de respuesta de Claude Code).

#### UC-005: Hints contextuales en skills existentes

**Descripción:** Añadir bloques explicativos condicionales en los skills /prd, /plan, /implement que se muestran las primeras 3 veces que un desarrollador ejecuta cada comando en un proyecto. Después de 3 usos, los hints se desactivan automáticamente.

**Pre-condiciones:**
- Skills existentes en .claude/skills/.
- Mecanismo de conteo de ejecuciones (puede usar .quality/logs/ o un counter file).

**Agentes involucrados:** Skills existentes (modificación), Orchestrator.

**AC-21:** La primera vez que se ejecuta /prd en un proyecto, se muestra un bloque de 3-4 líneas antes de iniciar que explica: qué es un PRD, qué formato usa el Engine (US/UC/AC), y qué es el Definition Quality Gate.

**AC-22:** La primera vez que se ejecuta /implement en un proyecto, se muestra un bloque que explica: qué es el Orchestrator, por qué no escribe código directamente, qué son los quality gates, y qué significa GO/NO-GO.

**AC-23:** Los hints se controlan mediante un fichero `.quality/hint_counters.json` que registra cuántas veces se ha ejecutado cada skill en el proyecto. Tras 3 ejecuciones del mismo skill, los hints dejan de mostrarse.

**AC-24:** Los hints no aparecen si el proyecto ya tiene más de 5 UCs completados (indicador de que el desarrollador ya conoce el flujo).

**AC-25:** Los hints no interfieren con el flujo del skill — se muestran antes de iniciar la ejecución real y no requieren confirmación del usuario para continuar.

#### UC-006: Wizard interactivo en onboard_project

**Descripción:** Mejorar la tool MCP `onboard_project` para que incluya un wizard interactivo que pregunte las opciones de configuración en lugar de requerir que el desarrollador conozca los parámetros de antemano.

**Pre-condiciones:**
- MCP Server corriendo.
- Proyecto nuevo sin .claude/project-config.json.

**Agentes involucrados:** MCP Server (tools/onboarding.py).

**AC-26:** Cuando se invoca `onboard_project` sin parámetros de configuración explícitos, la tool inicia un flujo interactivo que pregunta secuencialmente: (1) Nombre del proyecto, (2) ¿Usa Trello? → Si sí, board ID, (3) ¿Usa Stitch? → Si sí, project ID y modelo preferido, (4) ¿Qué VEG provider? → Canva/Freepik/Manual/Ninguno, (5) ¿Tiene tests existentes? → Para generar baseline.

**AC-27:** Cada pregunta incluye una explicación de una línea sobre qué es y para qué sirve la opción (ejemplo: "Trello se usa como pipeline de especificación para trackear User Stories y Use Cases").

**AC-28:** Si se invoca `onboard_project` con todos los parámetros explícitos, el wizard no se activa y el onboarding procede directamente como en v4.2.0 (retrocompatibilidad total).

**AC-29:** El wizard genera el project-config.json y lo escribe directamente en .claude/ del proyecto, informando al usuario de lo que se ha configurado.

**AC-30:** Si el usuario responde "no" a todas las opciones opcionales (Trello, Stitch, VEG), el onboarding genera una configuración mínima funcional y muestra un mensaje indicando que puede añadir integraciones después con `/optimize-agents`.

### US-03: Registry de Skills Contribuidos

**Como** desarrollador que trabaja en múltiples proyectos con requisitos verticales diferentes,  
**quiero** un mecanismo formal para instalar, descubrir y versionar skills específicos por dominio o stack sin modificar el core del Engine,  
**para** reutilizar conocimiento entre proyectos y compartir skills con otros desarrolladores.

**Prioridad:** P1  
**Estimación:** 20-28 horas  
**Impacto semanal:** Ahorro de 30-60 min por desarrollador + escalabilidad del ecosistema

---

#### UC-007: Estructura de skill externo con manifiesto

**Descripción:** Definir la estructura estándar de un skill externo (contribuido) que pueda instalarse en el Engine sin modificar el core. Incluye un manifiesto YAML con metadatos, dependencias, y reglas de auto-discovery.

**Pre-condiciones:**
- Engine v4.2.0+ instalado.

**Agentes involucrados:** Ninguno — este UC define la estructura, no la implementación.

**AC-31:** Un skill externo se estructura como un directorio con: SKILL.md (contenido principal con YAML frontmatter), manifest.yaml (metadatos del skill: nombre, versión, autor, stack/dominio compatible, dependencias de otros skills), y opcionalmente archivos de soporte (patterns.md, templates/, examples/).

**AC-32:** El manifest.yaml incluye campos obligatorios: name, version (semver), author, description, compatibility (lista de stacks con los que es compatible: flutter, react, python, gas, all), y campos opcionales: triggers (lista de keywords que activan auto-discovery), depends_on (lista de otros skills requeridos).

**AC-33:** La estructura de directorio sigue el patrón existente: `~/.claude/skills/{skill-name}/SKILL.md` para skills globales, o `.claude/skills/{skill-name}/SKILL.md` para skills de proyecto.

**AC-34:** Los skills externos conviven con los skills core del Engine sin conflictos de namespace. Si un skill externo tiene el mismo nombre que uno core, el core tiene prioridad y se emite un warning.

---

#### UC-008: Comando de instalación y listado de skills externos

**Descripción:** Mecanismo para instalar skills desde un directorio local o URL de repositorio Git, listarlos con su estado, y desinstalarlos.

**Pre-condiciones:**
- Estructura de skill externo definida (UC-007).

**Agentes involucrados:** install.sh (extensión), MCP tools.

**AC-35:** El script install.sh acepta un flag `--skill <path|git-url>` que instala un skill externo copiando su directorio a `~/.claude/skills/` (global) o `.claude/skills/` (local, con flag `--local`).

**AC-36:** La tool MCP `list_skills` (ya existente) incluye un campo `source` en su respuesta que indica si el skill es "core" (incluido en el Engine) o "external" (instalado posteriormente), junto con la versión del manifest.

**AC-37:** Un nuevo subcomando install.sh `--remove-skill <name>` desinstala un skill externo eliminando su directorio. No puede desinstalar skills core.

**AC-38:** Al instalar un skill externo, se valida que el manifest.yaml contiene todos los campos obligatorios. Si falta alguno, la instalación falla con un mensaje indicando qué campos faltan.

**AC-39:** Si el skill externo tiene campo `depends_on` en el manifest, la instalación verifica que las dependencias están disponibles (como skill core o externo ya instalado). Si faltan, emite un warning pero no bloquea la instalación.

---

#### UC-009: Auto-discovery de skills externos en pipeline

**Descripción:** Hacer que el Engine descubra y active automáticamente skills externos cuando sean relevantes para el contexto actual del /prd, /plan o /implement.

**Pre-condiciones:**
- Skills externos instalados (UC-008).
- Engine ejecutando un comando (/prd, /plan, o /implement).

**Agentes involucrados:** Orchestrator (lógica de discovery), Skills existentes.

**AC-40:** Cuando se ejecuta /prd, el Engine escanea todos los skills disponibles (core + externos) y activa aquellos cuyo campo `triggers` del manifest coincida con keywords presentes en la descripción del PRD o en el stack del proyecto.

**AC-41:** Un skill externo activado por auto-discovery inyecta su contenido como contexto adicional al sub-agente relevante. Para skills de compliance, se inyectan como AC adicionales en el PRD. Para skills de stack, se inyectan como patrones de arquitectura.

**AC-42:** El Orchestrator muestra una línea informativa cuando activa un skill externo: "Skill externo activado: {nombre} v{versión} — {descripción corta}". No requiere confirmación del usuario.

**AC-43:** Si un skill externo activado es incompatible con el stack del proyecto (campo `compatibility` del manifest no incluye el stack detectado), no se activa y se registra en los logs sin interrumpir el flujo.

### US-04: BDD como Módulo Exportable

**Como** desarrollador que recibe PRs de colaboradores que no usan el Engine completo,  
**quiero** poder ejecutar acceptance testing BDD (AG-09a/b) como módulo standalone contra cualquier PR que tenga AC definidos en un PRD,  
**para** garantizar que todo código mergeado cumple la especificación sin requerir el pipeline completo.

**Prioridad:** P0  
**Estimación:** 24-32 horas  
**Impacto semanal:** Ahorro de 1-2 horas + eliminación de 2-4 errores por PR no validada

---

#### UC-010: Skill /acceptance-check standalone

**Descripción:** Nuevo Agent Skill que ejecuta AG-09a (generación de .feature files) y AG-09b (validación con veredicto) contra una rama o PR específica, sin requerir que se haya ejecutado /implement previamente.

**Pre-condiciones:**
- PRD existente con AC definidos en doc/prd/.
- Rama o PR con código implementado.
- Stack del proyecto detectado.

**Agentes involucrados:** AG-09a (Acceptance Tester), AG-09b (Acceptance Validator).

**AC-44:** El skill /acceptance-check acepta como parámetro: (1) un UC-id que tiene AC definidos en un PRD, o (2) un US-id para validar todos sus UCs, o (3) una referencia a PR (número o rama).

**AC-45:** El skill localiza automáticamente el PRD correspondiente en doc/prd/, extrae los AC del UC indicado, y los pasa a AG-09a para generación de .feature files con step definitions.

**AC-46:** AG-09a genera los .feature files en un directorio temporal .quality/acceptance-check/{uc-id}/ sin modificar el directorio de tests del proyecto.

**AC-47:** AG-09b ejecuta la validación y emite un veredicto ACCEPTED/REJECTED/CONDITIONAL con evidence detallada: qué AC se cumplieron, cuáles fallaron, y por qué.

**AC-48:** El resultado se formatea como un bloque Markdown legible que puede copiarse directamente como comentario de PR en GitHub.

**AC-49:** Si se invoca con un PR number, el skill ejecuta `git diff main...{branch}` para identificar qué archivos cambió la PR y focaliza la validación en los AC relevantes a esos cambios.

---

#### UC-011: MCP Tools para acceptance check remoto

**Descripción:** Nuevas tools MCP que exponen la funcionalidad de acceptance-check para integración con CI/CD y herramientas externas.

**Pre-condiciones:**
- UC-010 implementado.
- MCP Server corriendo.

**Agentes involucrados:** MCP Server (nueva sección en tools/spec_driven.py o nuevo módulo tools/acceptance.py).

**AC-50:** Nueva tool MCP `run_acceptance_check` que acepta parámetros: project_path, uc_id (o us_id), branch (opcional, default: rama actual). Devuelve JSON con: verdict (ACCEPTED/REJECTED/CONDITIONAL), ac_results (array con cada AC y su status), feature_files (array con paths a los .feature generados), summary (texto legible).

**AC-51:** Nueva tool MCP `get_acceptance_report` que acepta un uc_id y devuelve el último reporte de acceptance generado para ese UC, si existe en .quality/evidence/.

**AC-52:** La tool `run_acceptance_check` registra cada ejecución en .quality/logs/ con los mismos campos que una sesión de /implement (timestamp, uc_id, verdict, duration), permitiendo tracking de acceptance checks independientes del pipeline.

**AC-53:** Si el UC indicado no tiene AC definidos en ningún PRD del proyecto, la tool devuelve un error descriptivo: "No se encontraron Acceptance Criteria para {uc-id} en doc/prd/. Verifica que el PRD existe y contiene AC para este UC."

---

#### UC-012: GitHub Action template para acceptance gate

**Descripción:** Template de GitHub Action que ejecuta /acceptance-check automáticamente en cada PR, usando el MCP Server del Engine como backend.

**Pre-condiciones:**
- UC-010 y UC-011 implementados.
- Repositorio con GitHub Actions habilitado.
- MCP Server accesible (local o remoto).

**Agentes involucrados:** Ninguno — es un template YAML.

**AC-54:** El template de GitHub Action (.github/workflows/acceptance-gate.yml) se activa en eventos pull_request (opened, synchronize, reopened) y ejecuta el acceptance check contra los UCs afectados por los cambios de la PR.

**AC-55:** La Action determina qué UCs están afectados comparando los archivos modificados en la PR contra la estructura de archivos documentada en los planes del proyecto (doc/plans/).

**AC-56:** El resultado de la Action se publica como comentario en la PR con el formato Markdown generado por AC-48. Si el veredicto es REJECTED, la Action falla (exit code 1), bloqueando el merge si el repositorio tiene branch protection configurado.

**AC-57:** La Action incluye variables de entorno configurables: `SDD_ENGINE_URL` (URL del MCP Server), `SDD_PROJECT_PATH` (path del proyecto), `SDD_FAIL_ON_CONDITIONAL` (true/false — si CONDITIONAL también bloquea el merge).

**AC-58:** El template se distribuye en templates/github-actions/acceptance-gate.yml del Engine e install.sh ofrece copiarlo al proyecto durante onboarding.

### US-05: Benchmarking Público con Datos Reales

**Como** potencial adoptante del Engine que ha accedido al repositorio,  
**quiero** ver métricas reales de proyectos implementados con el Engine,  
**para** evaluar su impacto real en productividad y calidad antes de decidir adoptarlo.

**Prioridad:** P2  
**Estimación:** 12-16 horas  
**Impacto:** Conversión de leads en el embudo de LinkedIn + credibilidad del Engine

#### UC-013: Generación automática de benchmark snapshot

**Descripción:** Nueva funcionalidad que genera un snapshot de métricas agregadas de todos los proyectos onboarded, anonimizando los datos sensibles, en formato Markdown publicable.

**Pre-condiciones:**
- Al menos 1 proyecto onboarded con datos de telemetría.
- Sala de Máquinas con datos.

**Agentes involucrados:** MCP Server (tools/telemetry.py), nuevo generador de snapshot.

**AC-59:** Nueva tool MCP `generate_benchmark_snapshot` que consulta la Sala de Máquinas y genera un archivo Markdown con métricas agregadas: número de UCs implementados (total y por proyecto anonimizado), coverage promedio, healing events y resolution rate, acceptance rate AG-09b, tiempo medio por UC, y delta count promedio (de US-01 si está implementado).

**AC-60:** Los nombres de proyectos se anonimizan como "Proyecto A", "Proyecto B", etc. Ningún dato que identifique al proyecto, cliente, o stack específico se incluye en el snapshot (excepto categorías genéricas: "Flutter app", "Python API").

**AC-61:** El snapshot incluye una sección de "Metodología" que explica cómo se calcularon las métricas, qué period cubre, y una nota de que son datos reales no simulados.

**AC-62:** El archivo se genera en docs/benchmarks/snapshot_{fecha}.md y se puede incluir opcionalmente en el README via include o link.

---

#### UC-014: Endpoint de métricas públicas en dashboard

**Descripción:** Nuevo endpoint REST en el dashboard de la Sala de Máquinas que sirve las métricas de benchmark en formato JSON para consumo externo (embeds en web, README badges, etc.).

**Pre-condiciones:**
- UC-013 implementado.
- Dashboard de Sala de Máquinas corriendo.

**Agentes involucrados:** MCP Server (dashboard_api.py).

**AC-63:** Nuevo endpoint GET `/api/benchmark/public` que devuelve las métricas del último snapshot en formato JSON con la misma estructura que el Markdown pero en campos tipados.

**AC-64:** El endpoint no requiere autenticación (es público por diseño) y tiene un rate limit de 60 requests/minuto.

**AC-65:** El endpoint incluye un campo `generated_at` con timestamp ISO 8601 y un campo `engine_version` con la versión del Engine que generó el snapshot.

**AC-66:** El endpoint devuelve status 404 con mensaje descriptivo si no hay ningún snapshot generado todavía.

## 3. Dependencias entre User Stories

```
US-01 (Spec-Code Sync) ──→ independiente, implementar primero
         │
         └──→ US-05 (Benchmarking) usa delta_count de US-01 como métrica
                    
US-02 (DX Onboarding) ──→ independiente, pero beneficia de US-03
         │
         └──→ US-03 (Skill Registry) permite skills externos en /quickstart

US-03 (Skill Registry) ──→ independiente

US-04 (BDD Exportable) ──→ independiente, pero la Action template (UC-012)
         │                   puede incluir acceptance-check en onboarding (US-02)
         │
         └──→ US-05 (Benchmarking) usa acceptance rate como métrica

US-05 (Benchmarking) ──→ depende parcialmente de US-01 y US-04 para métricas completas
```

## 4. Riesgos y Mitigaciones

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Los deltas de Spec-Code Sync son demasiado verbosos y ensucian el PRD | Media | Medio | Limitar a 500 tokens por fase (AC-02). Formato compacto predefinido. |
